# 60/30/10 Triage — Aplicado ao Workspace Alvo

Framework do Eduba (Jake Van Clief). Fonte completa: [files/vault-toolkit/constraints/06-layer-triage.md](../../../../files/vault-toolkit/constraints/06-layer-triage.md). Não duplicar a teoria aqui — este arquivo descreve **como aplicar o triage ao redesenhar o diretório alvo**.

## A regra

| Camada | % do esforço do workspace | Ferramenta |
|---|---|---|
| **60% Infraestrutura** | Tarefas determinísticas | Spreadsheets, databases, scripts simples, file system |
| **30% Orquestração** | Tarefas if/then | Zapier, Make, n8n, scripts de roteamento |
| **10% AI** | Julgamento / síntese / criatividade | LLM via Claude Code |

## Aplicação em stage 02 (Classify)

Para cada **modo de trabalho** detectado em stage 02 (escrita, produção, código, dados, etc.), aplicar o diagnóstico de 3 perguntas em ordem:

1. **Existe resposta determinística?** (cálculo, lookup, conversão, formatação) → camada 60. Não é trabalho de AI.
2. **Cabe em if/then com critérios conhecidos?** (roteamento, categorização, filtros) → camada 30. Não é trabalho de AI.
3. **Requer julgamento sobre informação não-estruturada?** → camada 10. Aqui AI agrega valor.

## Aplicação em stage 03 (Design)

Cada silo proposto deve declarar a distribuição esperada nos seus contratos L2:

```markdown
## Layer Distribution

- 60% (infra): pré-processamento de assets, file naming, indexação
- 30% (orq): roteamento de drafts por status, escolha de template por tipo
- 10% (AI): síntese de research, geração de primeira versão, review qualitativo
```

## Sinais de over-AI no diretório alvo

Detectar e flaggar no triage report quando o usuário está usando AI para o que não deveria:

| Sinal | Provável over-AI |
|---|---|
| Pasta `ai-prompts/` cheia de prompts para extrair dados estruturados | → deveria ser script ou query |
| Conversas salvas onde AI faz cálculos numéricos | → deveria ser planilha |
| Templates "preencha com AI" para campos com lista finita de opções | → deveria ser dropdown / regra |
| Workflow onde AI roteia entre N pastas baseado em palavra-chave | → deveria ser regex / Zapier |
| Pastas idênticas por cliente onde só nome muda | → deveria ser script de scaffold |

Quando detectado, o `triage-report.md` deve listar esses casos como **"AI candidates to demote"** — sugestão de mover para camada 60 ou 30.

## Sinais de under-AI

Inverso também é problema. Onde o usuário está fazendo manualmente o que AI faria bem:

| Sinal | Provável under-AI |
|---|---|
| Pastas com 50+ rascunhos sem nenhuma síntese | → AI pode resumir e indexar |
| Notas de reuniões em raw text sem ação extraída | → AI pode extrair action items |
| Documentos similares editados manualmente para cada cliente | → AI pode personalizar a partir de template |
| Pesquisa em PDFs sem cross-references | → AI pode mapear conexões |

Listar como **"AI candidates to adopt"** no triage report.

## Output esperado

A seção de triage do `triage-report.md` (stage 02) inclui:

```markdown
## 60/30/10 Triage

### Por modo de trabalho
| Modo detectado | Camada predominante | Justificativa |
|---|---|---|
| escrita | 10 (AI) | Síntese e voz exigem julgamento |
| produção | 60 (infra) + 10 (AI) | Build é determinístico, design é AI |
| distribuição | 30 (orq) | Roteamento por plataforma é regra |

### AI candidates to demote
- <pasta>: <razão>

### AI candidates to adopt
- <pasta>: <razão>
```

Stage 03 (Design) usa essa tabela para decidir **quais silos precisam de skills/MCPs** (camada 10) e **quais precisam de scripts/templates** (camadas 60/30) em vez de mais prompts.
